## 一、系统总体架构概览

### 1.1 研究背景与问题定义

在多智能体强化学习（MARL）中，奖励函数设计是决定策略质量的核心瓶颈。传统方法依赖人工设计，存在以下三大困境：

| 困境类型 | 具体表现 | 本项目的解决方案 |
|---------|---------|---------------|
| **初始零样本失真** | LLM 直接生成奖励函数时，因缺乏环境先验而产生"幻觉"（引用不存在的 API / 物理常量） | AIR 模块：强制 CoT 两阶段管线 |
| **非平稳收敛欺骗** | 仅以累计奖励为指标无法区分"真收敛"与"虚假震荡" | 多维收敛过滤拦截器 + 降级选拔算法 |
| **进化搜索发散** | 盲目变异导致搜索空间急剧膨胀，有效信息随代数快速稀释 | DREAM 自适应变异算子 |

### 1.2 整体数据流

```
用户执行 python run_evolution.py
  │
  ▼
RewardDesignAgent.step(generation)
  │
  ├─► generation == 0
  │     ├─► AIR 模块：阶段一 CoT 环境解析（低温度 T=0.3）
  │     └─► AIR 模块：阶段二 基于先验并行生成（T=1.0，n=4）
  │
  ├─► generation > 0
│     ├─► Reflection：诊断反思+算子分配（DREAM v2.0）
│     └─► DREAM：自适应变异（F1/F2/F3/L1 并发执行）
  │
  ├─► SimulationTool.run_parallel()
  │     ├─► SandboxManager：为每个候选创建独立训练沙盒
  │     └─► ParallelLauncher：并行 MADDPG 训练
  │
  ├─► LogAnalyzer.parse_logs()
  │     ├─► 提取多维指标（成功率 / 捕获时间 / 队形质量 / 碰撞惩罚）
  │     └─► 多维收敛过滤拦截器（式 5-7 判定）
  │
  ├─► analyze_results() → 降级选拔算法
  │     └─► EvolutionaryMemory.save() → JSON 持久化
  │
  └─► 输出：最优代码 + 反思 + 进化曲线
```

## 二、AIR 模块：基于思维链（CoT）的环境与任务分析

### 2.1 核心问题：初始零样本生成失真

当 LLM 在没有环境上下文的情况下直接生成奖励函数代码时，会产生两类典型的"幻觉失真"（Hallucination Distortion）：

**失真类型 A — 引用不存在的 API**  
LLM 可能生成如下代码（运行时崩溃）：

```python
# ❌ 幻觉代码：尝试访问不存在的属性
world.logic.is_collision(...)
world.adversary_params['max_speed']
```

**失真类型 B — 物理常量缺失**  
LLM 可能在计算中使用未定义的物理参数，导致奖励函数无法提供有效梯度引导。

### 2.2 AIR 模块的 CoT 两阶段管线设计

AIR 模块（**A**nalysis-**I**nformed **R**eward）通过强制 LLM 执行两阶段思维链，彻底解决上述失真问题。其核心思想是：**在生成代码之前，必须先建立对环境的物理级理解**。

#### 阶段一：强制环境与任务结构分析（CoT Prompt Engineering）

```python
# prompt_templates.py 第 164-216 行
def cot_analysis_prompt(task_description: str, env_context: Dict) -> str:
```

该阶段通过低温度采样（T=0.3）强制 LLM **严格按五个维度分步分析**：

```
阶段一 prompt 结构（强制五维度分解）
│
├── 维度 1：实现细节 (Implementation Details)
│     问题：环境代码使用了哪些依赖包？是否引入了外部未定义的变量？
│
├── 维度 2：环境结构 (Environment Structure)
│     问题：global_state 字典中包含哪些维度？物理意义是什么？
│           观测向量如何拼接？包含哪些相对信息？
│
├── 维度 3：智能体交互 (Agent Interactions)
│     问题：追捕者与逃跑者在 max_speed、size 等物理属性上有何异同？
│           "成功捕获"的数学判定条件是什么？
│
├── 维度 4：任务相关信息 (Task-relevant Information)
│     问题：围捕目标与哪些变量直接挂钩（正/负相关）？
│           怎样的空间拓扑代表"高质量包围圈"？
│
└── 维度 5：API 边界隔离 (API Boundaries)
      问题：哪些是伪类（CoreEnvLogic）不能实例化？
            物理常量必须如何声明？
```

**代码级约束注入**：阶段一 prompt 末尾明确写出"致命红线"约束，将 API 边界隔离（维度 5）作为硬约束固化在提示词中，从源头切除幻觉。

```python
# 阶段一 prompt 中的硬约束（prompt_templates.py 第 210-212 行）
"- CoreEnvLogic 类仅是文档伪代码，**绝非**运行时可实例化的真实类。"
"- 禁止调用 world.logic、world.adversary_params 等不存在的属性。"
"- 物理常量（如 size、max_speed、world_size）必须在函数内部硬编码。"
```

#### 阶段二：基于 CoT 先验的独立生成

阶段一产生的分析报告作为"条件先验"（Conditional Prior），通过高温度采样（T=1.0）驱动 LLM 并行生成 n=4 个候选奖励函数：

```python
# reward_design_agent.py 第 217-230 行
if generation == 0:
    # 阶段二：基于 CoT 先验并行生成候选代码
    prompt = self.prompt_builder.initial_generation_prompt_with_cot(
        self.task_description,
        self.env_context,
        self.cot_analysis_result   # ← 阶段一结果作为先验条件
    )
    raw_outputs = self.llm.generate(
        prompt=prompt,
        n=4,
        temperature=min(1.0, self.config['generation']['temperature'] + 0.2),
        ...
    )
```

### 2.3 关键设计决策

| 设计决策 | 理由 |
|---------|------|
| **阶段一低温度（T=0.3）** | 环境分析需要逻辑严密性与事实准确性，避免随机性引入错误先验 |
| **阶段二高温度（T=1.0）** | 奖励函数搜索需要足够的探索多样性，T=0.8+0.2=1.0 保证宽度 |
| **CoT 结果缓存机制** | 阶段一只执行一次并缓存至 `self.cot_analysis_result`，避免重复 API 调用开销 |
| **先验注入而非约束注入** | 不通过硬性规则禁止 LLM 做什么，而是通过"正确的环境理解"引导 LLM 自主生成合法代码 |

### 2.4 效果验证逻辑

阶段一分析完成后，LLM 对环境的理解被固化在 `cot_analysis_result` 字符串中，其中包含：
- 物理常量的具体数值（`world_size=2.5`，`capture_threshold=0.5`，`adversary_size=0.075`）
- API 边界的明确声明（哪些类不存在、哪些属性不可访问）
- 围捕成功的数学判定条件（欧氏距离 < 半径之和）

这些先验知识在阶段二生成时作为 context 注入，从根本上消除了零样本失真的温床。

## 三、多维度适应度评估机制

### 3.1 传统单维评估的局限性

传统 EA（遗传算法）类奖励函数进化方法通常以"累计奖励"（Cumulative Reward）作为唯一适应度指标。在多智能体环境中，这会导致两类问题：

1. **局部最优陷阱**：智能体可能学会"绕着目标兜圈子"而非真正围捕，累计奖励不低但任务完成率极低
2. **虚假收敛欺骗**：训练曲线末端的方差波动可能被误判为"已收敛"，实际上策略仍在震荡

### 3.2 多维适应度评估体系

本系统设计了**四维加权适应度函数**，在 `log_analyzer.py` 第 268-312 行实现：

$$
\text{Fitness} = w_1 \cdot \text{SuccessRate} + w_2 \cdot (-\text{CaptureTime}) + w_3 \cdot \text{FormationQuality} + w_4 \cdot (-\lvert\text{CollisionPenalty}\rvert)
$$

**权重配置**（`llm_config.yaml` 第 59-63 行）：

| 维度 | 权重符号 | 绝对值 | 物理意义 |
|------|---------|--------|---------|
| 成功率 (SuccessRate) | $w_1$ | **1.0** | 任务完成的核心判定 |
| 捕获时间 (CaptureTime) | $w_2$ | **-0.001** | 效率指标（越小越好，取负号转为越大越优） |
| 队形质量 (FormationQuality) | $w_3$ | **0.3** | 协同行为的间接代理指标 |
| 碰撞惩罚 (CollisionPenalty) | $w_4$ | **-0.5** | 安全性约束（取绝对值后取负号惩罚高碰撞） |

**数据管道**（`log_analyzer.py` 第 54-115 行）：

```
Sandbox 训练输出
  │
  ├─► training_log.json
  │     └─► _extract_task_performance() → success_rate, avg_capture_time
  │
  └─► reward_component_stats_*.json
        └─► _load_stats_file() → reward_components, collaboration_metrics
              └─► calculate_fitness() → 多维加权求和
```

### 3.3 多维收敛过滤拦截器（核心算法）

本系统严格参照 EUREKA 论文的方法论，在 `log_analyzer.py` 第 314-361 行实现了**三重收敛判别式**：

#### 判别式 1 — 增益判断（F_mean，式 5）

$$
F_{\text{mean}}: \quad J_{lm} > J_{em}
$$

**含义**：晚期均值（Late Mean）必须大于早期均值（Early Mean），保证训练过程中适应度确实在提升，而非原地踏步。

**代码实现**：

```python
# log_analyzer.py 第 341-349 行
early_data = episode_fitnesses[:alpha]    # 早期待窗口（默认前150回合）
late_data = episode_fitnesses[-beta:]      # 晚期待窗口（默认后300回合）

J_em = np.mean(early_data)  # 早期均值
J_lm = np.mean(late_data)   # 晚期均值

f_mean = bool(J_lm > J_em)   # 判别式5：J_lm > J_em
```

#### 判别式 2 — 收敛判断（F_std，式 6）

$$
F_{\text{std}}: \quad \frac{J_{lv}}{J_{ev}} < v_{\text{th}}
$$

**含义**：晚期方差（Late Variance）必须小于早期方差（Early Variance）的 $v_{\text{th}}$ 倍，保证策略已经稳定下来。

**设计考量**：$v_{\text{th}}=0.8$ 是本系统为多智能体环境特别调整的参数。相比 EUREKA 论文推荐的 0.5，0.8 放宽了方差收缩要求，原因是：

- 多智能体环境固有的非平稳性导致方差天然较大
- 过严的 $v_{\text{th}}$ 会导致大量有效策略被错误拦截
- 实践验证表明 0.8 在本任务上取得了更好的召回率-精度平衡

**代码实现**：

```python
# log_analyzer.py 第 343-350 行
J_ev = np.std(early_data)   # 早期方差
J_lv = np.std(late_data)     # 晚期方差
v_th = 0.8                   # 方差收缩容忍阈值

f_std = bool((J_lv / (J_ev + 1e-8)) < v_th)  # 判别式6
```

#### 判别式 3 — 趋势判断（F_slope，式 7）

$$
F_{\text{slope}}: \quad \text{cov}(t, \text{fitness}) > 0
$$

**含义**：适应度与时间的协方差必须为正，保证整体趋势是上升的，而非下降或无规律震荡。

**代码实现**：

```python
# log_analyzer.py 第 346-351 行
t = np.arange(L)  # 时间序列 [0, 1, 2, ..., L-1]
cov = np.cov(t, episode_fitnesses)[0][1]  # 协方差矩阵的(0,1)元素

f_slope = bool(cov > 0)  # 判别式7：协方差 > 0
```

#### 综合收敛条件

$$
\text{is\_converged} = F_{\text{mean}} \land F_{\text{std}} \land F_{\text{slope}}
$$

三个判别式必须同时满足，才认为策略真正收敛：

```python
# log_analyzer.py 第 349-353 行
f_mean = bool(J_lm > J_em)
f_std = bool((J_lv / (J_ev + 1e-8)) < v_th)
f_slope = bool(cov > 0)

is_converged = f_mean and f_std and f_slope
```

### 3.4 降级选拔算法（Degradation Selection Algorithm）

即使三重收敛过滤全部失败，系统也不会简单地放弃所有候选，而是采用**降级选拔算法**（`reward_design_agent.py` 第 401-428 行）：

```
降级选拔算法流程图

所有候选经过训练
      │
      ▼
┌─────────────────────────┐
│ 优先级1：完全收敛候选？    │── 是 ──► 取 Fitness 最高者
└─────────────────────────┘
      │ 否
      ▼
┌─────────────────────────┐
│ 优先级2：趋势上升候选？    │── 是 ──► 放宽 F_std 要求，取 Fitness 最高者
│   (f_mean ∧ f_slope)     │
└─────────────────────────┘
      │ 否
      ▼
┌─────────────────────────┐
│ 优先级3：全军覆没        │── ──► 取绝对 Fitness 最高者，强制下一轮突变
└─────────────────────────┘
```

**该算法的设计哲学**：多维收敛过滤是**拦截器**（interceptor），而非**否定器**（negator）。其目的是区分"好但未收敛"与"坏且未收敛"，而非粗暴地淘汰所有未收敛候选。

### 3.5 非平稳环境下的局部最优与虚假收敛缓解机制

多维适应度函数配合三重收敛拦截，共同构建了三层防护：

| 防护层级 | 作用对象 | 缓解的问题 |
|---------|---------|-----------|
| **第一层：队形质量 (FormationQuality)** | 局部最优 | 防止追捕者"搭便车"或只在边缘试探 |
| **第二层：碰撞惩罚 (CollisionPenalty)** | 虚假收敛 | 防止追捕者通过无意义碰撞刷奖励 |
| **第三层：三重收敛拦截** | 虚假收敛 | 区分真正的收敛与尾部噪声震荡 |

## 四、DREAM 模块：诊断反思驱动的自适应进化算子

### 4.1 从盲目变异到自适应分配的演进

在传统遗传算法中，变异算子通常是**无约束的随机翻转**（bit-flip mutation）或**高斯扰动**（Gaussian perturbation）。当直接应用于代码结构（如奖励函数）时，会产生两个严重问题：

1. **搜索空间发散**：随机扰动可能将原本有效的奖励结构彻底破坏，导致后代性能急剧下降
2. **信息稀释**：随着代数增加，早期有效的设计思想被随机性稀释，进化趋于随机搜索

EvoLeap  通过**语义级四向变异**取代随机翻转，从根本上约束搜索空间。但静态正交波束搜索（每代固定分配 `['F1', 'F2', 'F3', 'L1']`）仍存在 F2 算子效果不稳定、缺乏适应性等局限性。

DREAM  在此基础上引入**自适应算子分配机制**，让 LLM 根据训练结果自主选择下一代的算子组合。

### 4.2 DREAM 的四向受控变异算子设计

```
DREAM 四向变异算子
│
├── F1: Branch Augmentation（分支扩充）
│     约束：保留原代码，新增 1 个奖励分量
│     适用：LLM 诊断发现"缺少某类协同引导"
│
├── F2: Component Reconstruction（分量重构）【v2.0 重定义】
│     约束：重构失效分量的数学表达（线性→指数、引入平滑阈值等）
│     适用：LLM 诊断发现"某分量持续为零或起反作用"
│     策略变化：从"破坏性删除"改为"保守性重构"，保留物理维度信息
│
├── F3: Equilibrium Tuning（平衡微调）
│     约束：不增不减，仅修改各分量权重系数
│     适用：LLM 诊断发现"权重比例失调"
│
└── L1: Paradigm Leap（范式跃迁）
      约束：从零构建全新奖励函数
      适用：LLM 诊断发现"整体设计思路失效"
```

### 4.3 自适应算子分配机制

DREAM 的核心创新在于**让 LLM 参与算子分配决策**，而非静态循环分配：

```
自适应分配流程

训练日志 → Reflection（诊断+分配） → 正则解析 → 基数校验 → 并发执行
                                      │
                                      ├── 通过 → 使用 LLM 分配的算子
                                      │
                                      └── 失败 → 回退标准正交分配
```

**基数约束**：每种算子最多分配 2 次，防止 LLM 输出格式错误时导致搜索空间塌陷。

### 4.4 DREAM 的并发执行与反射反馈机制

#### 并发自适应变异

```python
# reward_design_agent.py 第 289-327 行
operators = ['F1', 'F2', 'F3', 'L1']
assigned_operators = [operators[i % 4] for i in range(n_candidates)]  # 兜底

# 从 reflection 中解析算子分配
matches = re.findall(r'Candidate \d+:\s*(F1|F2|F3|L1)', reflection, re.IGNORECASE)
if len(matches) >= n_candidates:
    assigned_operators = [m.upper() for m in matches[:n_candidates]]
    # 基数约束校验
    counts = Counter(assigned_operators)
    is_valid = all(v <= 2 for v in counts.values())
    if not is_valid:
        assigned_operators = []  # 触发兜底

# 并发执行
with concurrent.futures.ThreadPoolExecutor(max_workers=n_candidates) as executor:
    ...
```

每代使用 `ThreadPoolExecutor` 并发提交多个方向的变异任务，充分利用 LLM 调用的 I/O 等待时间。

#### 诊断反思升级（Reflection v2.0）

DREAM 的 Reflection 阶段从"仅输出病症诊断"升级为"诊断 + 算子分配方案"：

```python
# prompt_templates.py 第 145-170 行
prompt = """...
# 任务要求
1. **病理诊断**：简述哪些分量起主导作用，哪些失效...
2. **算子分配**：基于诊断，为下一代的4个候选独立分配算子。
   - **硬性约束**：【每种算子最多只能被选择 2 次】。

# 强制输出格式
[病理诊断]
(你的诊断内容，限200字)

[算子分配]
Candidate 0: <F1/F2/F3/L1>
Candidate 1: <F1/F2/F3/L1>
Candidate 2: <F1/F2/F3/L1>
Candidate 3: <F1/F2/F3/L1>"""
```

**关键设计**：Reflection prompt 剥夺了 LLM 的**完全重构决策权**（仍然禁止输出具体代码修改方案），但赋予了 LLM **算子选择权**。这使得 LLM 可以根据训练结果动态调整探索方向，而非机械地执行固定模式。

### 4.5 搜索空间稳定性的数学保证

DREAM 通过四个层次保证搜索空间的稳定性：

| 层次 | 机制 | 保证的稳定性类型 |
|------|------|---------------|
| **算子层约束** | 每个算子有明确的"不变式"（Invariant） | 结构稳定性：保证变异不破坏已有有效结构 |
| **基数约束** | 每种算子最多分配 2 次 | 搜索多样性：防止单一算子垄断搜索空间 |
| **反馈层约束** | Reflection 强制 LLM 输出"病症+分配" | 信息稳定性：避免诊断与处方之间的语义冲突 |
| **选拔层约束** | 降级选拔算法优先选择收敛候选 | 性能稳定性：保证每代的最优性能不低于父代 |

### 4.6 算法伪代码（Algorithm: DREAM Operator）

```
Algorithm 1: DREAM Operator
Input: parent_code, reflection, n_candidates
Output: assigned_operators (list of operator types)

1  assigned_operators ← []
2  matches ← RegexFind(reflection, "Candidate \d+:\s*(F1|F2|F3|L1)")
3  
4  if len(matches) ≥ n_candidates then
5      assigned_operators ← [m.upper() for m in matches[:n_candidates]]
6      counts ← Counter(assigned_operators)
7      is_valid ← ∀v ∈ counts.values(): v ≤ 2
8      
9      if ¬is_valid then
10         Print "LLM violates cardinality constraint {counts}, using fallback"
11         assigned_operators ← []
12     else
13         Print "Successfully parsed adaptive operators: {assigned_operators}"
14 else
15     Print "Failed to parse operators from reflection, using fallback"
16
17 if not assigned_operators then
18     operators ← ['F1', 'F2', 'F3', 'L1']
19     assigned_operators ← [operators[i mod 4] for i in range(n_candidates)]
20
21 return assigned_operators
```

3. **强调 DREAM 的自适应搜索理念**：与 EvoLeap 的静态正交搜索相比，DREAM 让 LLM 根据训练结果动态调整算子分配，实现"诊断驱动的自适应变异"。
4. **重点阐述 F2 范式转变**：解释为什么将"删除失效分量"改为"重构失效分量"是符合 MARL 特性的设计决策。
5. **说明三者协同的正反馈循环**：AIR 提供起点 → DREAM 提供进化动力 → 多维评估提供质量门控 → Memory 保留历史最优 → 下一代 DREAM 以更好的父代为基础继续进化。
